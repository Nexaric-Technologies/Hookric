# Security Policy

HookRick is a developer tool. Its security model is built on three
principles:

1. **No server-side persistence.** The capture server (Worker or Express)
   stores nothing. Every request is broadcast to its subscribers and
   forgotten.
2. **Per-endpoint isolation.** Each endpoint ID maps to its own Durable
   Object instance. There is no shared in-memory state between endpoints.
3. **Client-side secret masking.** Authorization, API keys, and cookies are
   masked in the UI by default. Raw values are still in LocalStorage /
   IndexedDB for replay, which is the same trust model as Postman.

## Supported versions

| Version | Supported          |
|---------|--------------------|
| `main`  | :white_check_mark:  |
| < 1.0   | :x:                |

HookRick is currently pre-1.0. There are no tagged releases yet — every
deploy is from `main`.

## Reporting a vulnerability

**Please do not file a public issue for security bugs.**

Email **security@nexaric.com** with:

- A short description of the vulnerability
- Steps to reproduce (preferably a `curl` we can run)
- The impact you observed or expect
- Your handle, if you'd like public credit in the fix commit

We will:

- Acknowledge within **2 business days**
- Triage and reproduce within **5 business days**
- Ship a fix or mitigation as soon as practical, coordinated with you
  on disclosure timing

## Scope

In scope:

- Capture pipeline (Worker + Durable Object, Express server)
- Realtime broadcast (WebSocket inside the DO, Socket.IO in the dev server)
- Client-side storage and replay (LocalStorage / IndexedDB)
- PWA service worker, notification permission flow, payload shape
- Push notification opt-in and unsubscribe

Out of scope (please report to the upstream vendor):

- Cloudflare Workers / Durable Objects platform bugs
- Cloudflare Pages hosting issues
- npm / Node.js runtime issues
- Browser bugs in the Web Push API or Notification API

## Endpoint-ID entropy

An endpoint ID is 8 base62 characters (`[0-9A-Za-z]{8}`) generated by
`crypto.getRandomValues`. That's ~47 bits of entropy, which is enough to
make URL guessing economically irrational for the lifetime of the request
capture window. If you need higher entropy (e.g. for a long-lived secret
URL), rotate the endpoint often and consider URL-signing at the proxy.

We do not enforce rate limiting on the capture endpoint — it's meant to be
unrestricted. If you operate a public-facing instance and want a soft
limiter, fork the Worker and add a token bucket per ID.

## Tenant isolation

In the production Cloudflare deployment, every endpoint ID is its own
Durable Object instance (`idFromName(endpointId)` in `worker/index.js`).
A request to `/r/hook-agent/foo` only touches the `foo` DO. A bug or
DoS in one DO cannot read from, or starve, another.

The local Express server uses in-memory `Map<endpointId, Set<socket>>`
with no cross-endpoint coupling — same guarantee, different shape.

## Data retention

- **Server**: zero retention. A request is parsed, broadcast, and dropped
  in the same event loop tick.
- **Browser**: persisted in LocalStorage (metadata, up to 200 requests) and
  IndexedDB (body blobs, up to 50 MB total). User-cleared on "Clear all".
- **Push notifications**: the Notification API shows them via the OS; the
  browser owns the retention. We never store notification history.
